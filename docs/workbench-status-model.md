# Workbench Status Model

这份说明定义这套 assistant workbench 如何表达状态，避免把“有脚本”误报成“系统稳定”。

## Status Levels

- `已验证`
  - 真实执行过，且有结果物、日志、ack 或检查输出来支撑
- `部分可用`
  - 主链路存在，但稳定性、输入边界、故障恢复或验收还不完整
- `待整理`
  - 能力或脚本存在，但入口、依赖、验证方式还没系统化
- `仅设想`
  - 方向明确，但还没有形成稳定实现

## Delivery Layers

对于自动化链路，默认至少拆成以下层：

1. `生成`
2. `发送`
3. `定时触发`
4. `ack`
5. `客户端可见性`

规则：
- 任一层成功，不等于整条链路成功
- `发送成功` 不能自动推导为 `客户端可见`
- `历史曾经成功` 不能自动推导为 `当前方案稳定`

## Response Labels

对外汇报默认优先使用：

- `已完成`
- `候选`
- `低置信度`
- `待验证`

## Evidence Rule

任何结论都尽量落在证据上：

- 文件证据：结果文件、状态文件、配置文件
- 日志证据：`sent`、`ack`、`skip`、`error`
- 验证证据：最小诊断脚本、结构检查、dry-run 输出

没有证据时，应明确说“不确定”，而不是补全叙事。
