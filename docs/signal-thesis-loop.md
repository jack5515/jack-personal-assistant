# Signal Thesis Loop

这份文档定义 `巡检 -> 证据 -> thesis -> 决策 -> 动作 -> 长期记忆 -> 下一轮巡检` 的闭环。

目标：
- 不让巡检只停留在资讯层
- 不让长期记忆只停留在结论仓库
- 让外部 signal 和内部判断持续互相修正
- 让每一轮巡检都能逼近创业收敛，而不是堆观点

## Loop

1. `巡检`
- 从用户痛点、竞品定价/帮助中心、社区抱怨、官方更新里抓 signal
- 只保留会改变判断的高价值内容

2. `证据`
- 先判断证据等级：已验证 / 候选 / 低置信度 / 待验证
- 默认优先看用户痛点、付费摩擦、工作流断裂、行为变化、反例

3. `thesis`
- 再往上一层，写出更长期的模式判断
- 例如：楔子该选哪里？谁更可能付费？产品是在走向服务系统，还是在滑向 AI 外包？

4. `决策`
- 每条高价值 signal 默认回答：它验证了什么、削弱了什么、要求本周做什么、目前不该做什么
- 如果回答不到这里，这条内容大概率还只是资讯

5. `动作`
- 把高价值内容压成 `约谁聊 / 补什么证据 / 打什么 MVP / 改什么销售口径 / 停什么方向`
- 没有动作的判断，默认价值打折

6. `长期记忆`
- 只有值得复用的 thesis、框架和产品判断，才进入 `MEMORY.md` 或正式文档
- 不是每条巡检都直接沉淀

7. `下一轮巡检`
- 从 thesis 反推新的 watchlist
- 下一轮优先盯哪些公司、哪些功能、哪些用户反馈、哪些验证信号

## Practical Rule

每次强 signal，默认问 `5` 个问题：

1. 这条 signal 说明什么更长期的变化？
2. 它会不会改变当前某条 thesis？
3. 它验证了什么，又削弱了什么？
4. 它要求 Jack 本周做什么动作？目前不该做什么？
5. 下一轮巡检该继续盯什么来验证或推翻它？

如果当天没有强 signal，也默认问：
- 今天最该补哪条反证？
- 哪个候选机会值得拆一层？
- 当前最缺哪类证据？

## Failure Modes

最常见的失败方式有 3 种：

- `只有新闻，没有 thesis`
- `有 thesis，但没有动作`
- `无强 signal 就空转`
- `有 thesis，但没有回流到下一轮 watchlist`

## Minimal Upgrade Path

如果不想一步做太重，最小升级也至少要做到：

- 巡检简报里明确写 `影响哪条 thesis`
- 对特别强的 signal，补 `创业动作 / 不做什么`
- 没有强 signal 时，补 `反证 / 候选机会拆解 / 关键证据缺口`
- 每隔几天回看一次，把重复出现的 thesis 提炼进长期文档
