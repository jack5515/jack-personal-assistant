# AI Watchlist

这份文档定义接下来一轮巡检应该重点盯什么，避免 AI 巡检退回泛扫描，或滑回“高质量行业新闻整理”。

目标：
- 让巡检更像有方向的雷达
- 让 thesis 反向塑造 signal 搜索范围
- 降低普通新闻占比，提高高价值 signal 命中率
- 让每一轮巡检更接近创业楔子、用户、付费点和 kill list

## Current Watchlist

### 1. 用户真实痛点证据
重点盯：
- GitHub issues / discussions 里的高频抱怨
- Reddit / X / App Store 差评里重复出现的弃用原因
- 竞品社区里“哪里最卡、最烦、最容易放弃”
- 用户明确写出“愿意换工具 / 已经换工具 / 终于找到替代”的证据
- 会议后跟进、截图即办、跨端接力、待办推进、知识沉淀等场景里的真实抱怨

为什么盯：
- 这是离创业最近的证据层。没有强痛点，后面的包装、产品和收费都站不住。

### 2. 付费与包装证据
重点盯：
- 定价页、套餐拆法、免费额度、积分逻辑
- 升级触发点和限制条款
- 竞品在卖“模型能力”还是卖“服务结果”
- 哪些功能开始被放进收费墙：记忆、搜索、浏览器、长期执行、协作、恢复
- 帮助中心、账单规则、暂停 / 恢复 / 自动续费口径

为什么盯：
- 真实付费结构比品牌叙事更能说明市场愿意为什么买单。

### 3. 工作流断裂证据
重点盯：
- 会议结束后谁来跟进、谁来推进、哪里断掉
- 截图之后怎么变成任务、知识或行动
- 手机、桌面、IM、浏览器之间怎么接力
- 待办系统怎么从“记录”变成“推进”
- 知识沉淀怎么从“存了”变成“下次自动可用”
- 状态跟踪、失败恢复、权限审批、自动重试哪里最缺

为什么盯：
- 这条线最接近“强闭环楔子”。没有真实断裂点，就没有必要做产品闭环。

### 4. 行为变化证据
重点盯：
- 用户是否开始把 AI 当执行者，而不是回答者
- 用户是否主动要求状态、恢复、追踪、交付
- 用户是否更在意“少漏事、少折腾、自动推进”
- 大厂是否在教育执行型心智，但还没真正解决体验问题
- 回答型产品是否开始被用户嫌“不够用”“还要自己收尾”

为什么盯：
- 这决定 Jack 做的是小改良，还是 category shift。

### 5. 反例证据
重点盯：
- 热门 agent 产品背后是不是人工重兜底
- 失败后是否只能靠客服或人工救火
- 是否每个客户都像定制项目，能力没有沉淀进系统
- 看起来很火，但留存、付费、口碑很差的案例
- 模型一升级就失去差异化的壳层产品

为什么盯：
- 这是 kill list 的来源。早点确认哪些路不该走，比多看几条热点更值钱。

## Exclusions

当前默认降权：
- 普通融资新闻
- 没有明确产品动作的泛观点文章
- 单纯“某模型更强了”但没有产品承接的更新
- 只有官方博客、没有用户痛点或付费证据的发布稿
- 只会强化“大厂很强”，但不能转成创业动作的新闻
- 没有公开证据链的社区传闻

## Update Rule

每次强 signal 后，默认问：
- 这条内容挂到哪条 thesis？
- 它增强了什么，还是削弱了什么？
- 它会不会替换掉当前某条低价值 watch item？
- 它要求 Jack 本周做什么动作？
- 它说明目前不该做什么？
