# Judgment Frameworks

这份文档用于把长期背景知识从“记了很多结论”升级成“可复用的判断框架”。

目标不是再堆一份背景材料，而是明确：
- 我们默认如何判断一件事
- 一条 signal 要经过哪些问题过滤
- 哪些标准会反复影响后续巡检、产品分析和创业判断

## Use Rule

后续遇到新的 signal、产品方向、竞品动作或创业机会时，优先尝试映射到下面这些框架。

不要只记：
- 我们当前结论是什么

还要记：
- 我们为什么这么判断
- 哪个条件变化会推翻这个判断
- 下一轮该补什么证据

## Framework 1: Signal -> Thesis

一条巡检 signal 默认按这 4 步处理：

1. `发生了什么`
2. `为什么这不是普通新闻`
3. `这反映的长期模式是什么`
4. `这会不会改变 Jack 的产品/创业判断`

如果只能回答前 1-2 步，这条内容更像资讯，不够像 thesis。

## Framework 2: Product Opportunity

评估一个产品机会时，默认看：

- `真实需求强度`
- `频次`
- `用户是否愿意持续付费`
- `结果是否可验证`
- `是否能沉淀成系统，而不是人工外包`
- `是否有入口、记忆、流程或数据壁垒`

### 常见反问

- 这是一次性新鲜感，还是长期高频需求？
- 这是“看起来能做”，还是“用户真的会反复回来用”？
- 价值来自系统，还是来自大量人工兜底？
- 如果基座模型继续进步，这个层的价值会被压缩吗？

## Framework 3: Agent Product

评估 Agent 路线时，默认拆成三层：

- `通用 runtime / shell / OS`
- `垂类 workflow / skill`
- `结果交付 / 服务体验`

默认重点不是“能不能做很多事”，而是：

- 是否稳定
- 是否可恢复
- 是否有记忆
- 是否有权限和状态可视化
- 是否真的形成了 workflow 闭环

## Framework 4: Service vs Outsourcing

凡是讨论 Claw / Agent 商业化，默认都要过这条红线：

- 核心价值来自系统默认配置和长期服务能力
- 还是来自背后有人不断替用户手工完成

如果更像后者，就要警惕滑向 AI 外包。

## Framework 5: AI PM / Product Judgment

默认判断 AI 产品或 AI PM 能力时，重点看：

- 是否定义了成功标准与评测
- 是否理解模型、工程、商业 trade-off
- 是否设计了 human-in-the-loop
- 是否能沉淀 bad case 与反馈闭环
- 是否把方法论沉淀成 AI 可复用的流程/知识/技能

## Framework 6: Next-Round Watchlist

每条高价值 signal 最后尽量补一句：

- `下一轮巡检要继续盯什么`

目的不是凑格式，而是让长期判断反过来塑造下一轮巡检方向。

## What Good Looks Like

好的背景知识不只是：
- 存很多结论

而是：
- 能指导下一次怎么判断
- 能解释为什么这次判断成立
- 能指出哪些证据一变，结论就该改
