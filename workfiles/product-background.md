## 当前核心结论

- 当前产品不是单纯的 AI 录音硬件，也不是单一 Chatbot，而是一个以 `阶跃AI App + PC + Web + 效率硬件路线` 为载体的多端 Agent 产品体系。
- 短期切入口是 `录音与效率场景`，中期目标是 `让普通用户也能低门槛获得 Agent 能力`，长期目标是在 `context -> memory -> execution -> ecosystem` 这条链路上占住生态位。
- 前端负责 `Context Capture / Context Import / Agent Connection`，中层负责 `Memory / Knowledge Base / Reading`，后端负责 `Agent Execution / Skill Ecosystem / Commercialization`。
- 当前最接近的外部类比可更新为：`Plaud + NotebookLM + OpenClaw + Skill Store`。
- 录音硬件不是终局，真正要争夺的是“全天候记录用户 context，并通过 Agent 自动处理 context 帮用户完成工作”的产品和生态位。

## 待验证问题清单

- 普通用户真正愿意为哪一类 Agent 交付能力持续付费：录音整理、知识问答、自动执行，还是跨端协同？
- `App / PC / Web` 三端的核心职责是否已经足够清晰，是否会发生能力重叠和用户认知混乱？
- 录音硬件作为短期切入口，能否自然过渡到更强的 Agent 产品心智，而不是被用户固化为单点工具？
- `StepClaw / 水产市场 / 本地部署` 这条生态链路，对非技术用户的可理解性和可转化性到底有多强？
- 面向普通用户时，产品最大障碍到底是 Agent 能力不足，还是交互、引导、安全和冷启动成本过高？
- 产品最终的商业化主引擎更可能来自硬件销售、订阅服务、技能生态分成，还是企业化方案？

## 产品愿景

### 已确认

- 文档的核心判断是：AI 效率硬件真正争夺的生态位，不是录音本身，而是 `全天候记录用户所有 context，并打通工具软件，让 AI 自动处理 context 帮用户完成工作`。
- 长期目标不是卖一个录音笔，而是做成 `高度有用且个性化的终端 + 云端 AI 助手`。
- 新补充文档进一步明确：产品形态要从 Chatbot 演进到 Agent，以打通人机交互的最后一公里，并在 Agent 时代尽快取得自己的生态位。

### 当前判断

- 这个产品的终局不是“记录工具”，也不是“又一个聊天框”，而是“个人工作记忆层 + 跨端行动层”。
- 录音硬件只是最强的 context 入口之一，真正的护城河应该来自：
  - 更完整的 context 积累
  - 更强的行业知识理解
  - 更可靠的 Agent 交付能力
  - 技能市场与触发器带来的生态位
  - 跨端、跨场景、跨时间的持续记忆

## 战略机会窗口与生态位

### 已确认

- 新文档提出两个重要机会窗口：
  - `打通人机交互的最后一公里`
  - `在以 OpenClaw 为代表的 Agent 时代大潮里取得自己的生态位`
- 文档明确判断：OpenClaw 可能会成为一个跨 `iOS / Android / Windows / Mac / Linux` 的新生态，类似 `Agent 时代的 Android`。
- 文档进一步给出生态判断：
  - `Skill` 可能会像未来的 App
  - `Skill Store` 可能会像未来的 App Store
  - `水产市场` 会像新的 Playground，并形成自己的商业生态
- 文档也指出当前阶段的现实约束：
  - OpenClaw 仍偏专业和偏 Chat 交互
  - 要继续向更大规模用户破圈，`LUI` 和更广泛的 Skill 覆盖仍是关键
- 产品形态被明确要求做整体规划：
  - `阶跃AI App` 侧重硬件连接和感知，提供 Agent 连接
  - `阶跃AI PC` 侧重 Agent 能力和生态
  - `阶跃AI Web` 侧重 Browser 快速体验，提供 Agent 连接
- 文档建议把公司品牌、模型、业务的增长和营销势能统一打到 `阶跃AI` 这个品牌上，用软件产品承接用户，再反哺硬件销售。

### 当前判断

- 阶跃 AI 的“第二个机会窗口”不只是继续做 AI 录音，而是借 OpenClaw 浪潮，把产品从 `Chatbot 产品` 升级成 `Agent 产品`。
- 阶跃 AI 不需要和 OpenClaw 社区在底层技术上做一比一竞争，更现实的差异化应该来自：
  - 面向普通用户的产品化包装
  - 本地与云端的一键部署
  - 更好的安全、权限可视化和修复能力
  - 更清晰的场景表达与冷启动引导
  - 与硬件、录音、知识库的更深结合
- 从战略上看，真正稀缺的不是再做一个 Agent 外壳，而是打通 `context 积累 -> 记忆沉淀 -> Agent 执行 -> 市场生态 -> 商业化` 的连续链路。

## 品牌与用户心智

### 已确认

- `阶跃AI App / PC / Web` 被要求统一品牌和 Logo。
- Agent 的昵称与桌面版对齐为 `小跃`。
- `小跃` 的 IP 方向被定义为一颗散发暖光的小星星，用于把抽象 AI 技术人格化，强化“温暖、可靠、陪伴式 AI 伙伴”的心智。

### 当前判断

- 品牌统一不是视觉层小修小补，而是为了让 `App / PC / Web / Agent / 硬件` 被用户理解成同一个产品体系。
- `小跃` 作为 Agent 形象有利于跨场景透传，但它只能放大已经成立的价值，不能替代场景本身的清晰度。

## 目标用户

### 已确认

原始文档明确瞄准以下高强度会议和沟通人群：

- 科技自媒体
- 创投人士
- 律师
- 科技公司白领
- 科技行业猎头

这些人群的共性是：

- 高频会议与访谈
- 高价值信息密度高
- 会后整理耗时
- 对专业名词、行业 context、后续跟进和知识沉淀有更高要求

新文档反映出的另一类早期用户还包括：

- AI 爱好者
- 开发者
- AI 圈 KOL
- 愿意尝试本地 Agent、技能市场、触发器生态的种子用户

### 当前判断

- 当前用户结构可以分成三层：
  - `核心付费用户`：高频沟通和高价值 context 人群，为结果付费
  - `生态种子用户`：开发者、AI 爱好者、KOL，为可玩性、扩展性和社区资产付费或贡献
  - `潜在大众用户`：不想研究技术，只想低门槛获得 Agent 能力的人
- 这意味着产品需要同时处理两条增长逻辑：
  - 用 `StepClaw / 水产市场 / 本地部署` 吸引生态和技术用户
  - 用 `录音 / 知识库 / Agent 交付` 承接高价值效率用户

## 核心场景

### 已确认

文档里反复出现的核心场景包括：

- 线下会议录音
- 手机通话录音
- 蓝牙耳机电话录音
- PC 会议录音，尤其是飞书、腾讯会议、钉钉
- 突发性的语音备忘
- 多语言或跨国会议
- 会后纪要、报告、PPT、知识卡片生成
- Todo 提取与跟踪
- 跨项目、跨案件、跨客户的知识检索和比对
- 一键部署 OpenClaw / StepClaw
- 在 App / Web / PC 上管理 Agent、技能和资产
- 用 `Channel`、`触发器`、技能市场把 Agent 变成更主动、更可玩的体验
- 在 App 上连接 PC Agent、OS Agent 和未来硬件入口

### 当前判断

- 这些场景现在不该只被抽象成 `采集 -> 理解 -> 执行` 三段，而应该升级成四段：
  1. `采集`
  2. `理解`
  3. `执行`
  4. `生态扩展`
- 对应产品能力就是：
  1. 极致录音和转写
  2. context 知识库和专业理解
  3. Agent 自动交付和跨端连接
  4. Skill / Channel / 触发器 / 市场带来的持续扩展

## 产品形态与三端分工

### 已确认

原始文档已提出的硬件形态包括：

- 类 Plaud 卡片形态
- `Plaud in Phone`
- `PC + Hidock`
- `Plaud + 蓝牙`
- 中期的 InPhone 形态，即通过 Agent 手机修改音频架构和音频芯片支持远场拾音

新文档进一步把软件和多端产品形态明确为：

- `阶跃AI App`
- `阶跃AI PC`
- `阶跃AI Web`
- 专用硬件版
- 手机定制版
- 车机定制版
- Code CLI

三端分工被直接定义为：

- `App`：连接模型、连接 Context、连接 Agent、连接人和 Agent 共生的场景
- `PC`：Cowork、OpenClaw、Skill 生态、Context
- `Web`：快速体验

另一个关键信号是：

- `Clawphone` 被定义为“软件先行，从录音切入，再扩展更多 AI Agent 手机场景”

### 当前判断

- `App` 是最广泛用户入口，也是未来的硬件连接器、OS 连接器和资产管理入口。
- `PC` 是当前最适合承接本地 StepClaw、生产力执行、技能市场和生态玩法的主战场。
- `Web` 更像拉新和快速试用入口，不是核心护城河，但对转化很重要。
- `专用硬件 / 手机定制版 / 车机定制版` 是中长期护城河放大器，但它们应该建立在已经成立的软件链路之上，而不是反过来要求软件先为硬件背书。
- 硬件路线本身也出现了阶段变化：
  - 短期仍可用卡片形态切入
  - 中长期更重视 InPhone / Clawphone 这类系统级入口
