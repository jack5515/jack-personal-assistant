## 当前核心结论

- 当前产品不是单纯的 AI 录音硬件，也不是单一 Chatbot，而是一个以 `阶跃AI App + PC + Web + 效率硬件路线` 为载体的多端 Agent 产品体系。

## 产品愿景

### 已确认

- 文档的核心判断是：AI 效率硬件真正争夺的生态位，不是录音本身，而是 `全天候记录用户所有 context，并打通工具软件，让 AI 自动处理 context 帮用户完成工作`。
- 长期目标不是卖一个录音笔，而是做成 `高度有用且个性化的终端 + 云端 AI 助手`。
- 新补充文档进一步明确：产品形态要从 Chatbot 演进到 Agent，以打通人机交互的最后一公里，并在 Agent 时代尽快取得自己的生态位。

### 当前判断

- 这个产品的终局不是“记录工具”，也不是“又一个聊天框”，而是“个人工作记忆层 + 跨端行动层”。
- 录音硬件只是最强的 context 入口之一，真正的护城河应该来自：
  - 更完整的 context 积累
  - 更强的行业知识理解
  - 更可靠的 Agent 交付能力
  - 技能市场与触发器带来的生态位
  - 跨端、跨场景、跨时间的持续记忆

## 战略机会窗口与生态位

### 已确认

- 新文档提出两个重要机会窗口：
  - `打通人机交互的最后一公里`
  - `在以 OpenClaw 为代表的 Agent 时代大潮里取得自己的生态位`
- 文档明确判断：OpenClaw 可能会成为一个跨 `iOS / Android / Windows / Mac / Linux` 的新生态，类似 `Agent 时代的 Android`。
- 文档进一步给出生态判断：
  - `Skill` 可能会像未来的 App
  - `Skill Store` 可能会像未来的 App Store
  - `水产市场` 会像新的 Playground，并形成自己的商业生态
- 文档也指出当前阶段的现实约束：
  - OpenClaw 仍偏专业和偏 Chat 交互
  - 要继续向更大规模用户破圈，`LUI` 和更广泛的 Skill 覆盖仍是关键
- 产品形态被明确要求做整体规划：
  - `阶跃AI App` 侧重硬件连接和感知，提供 Agent 连接
  - `阶跃AI PC` 侧重 Agent 能力和生态
  - `阶跃AI Web` 侧重 Browser 快速体验，提供 Agent 连接
- 文档建议把公司品牌、模型、业务的增长和营销势能统一打到 `阶跃AI` 这个品牌上，用软件产品承接用户，再反哺硬件销售。

### 当前判断

- 阶跃 AI 的“第二个机会窗口”不只是继续做 AI 录音，而是借 OpenClaw 浪潮，把产品从 `Chatbot 产品` 升级成 `Agent 产品`。
- 阶跃 AI 不需要和 OpenClaw 社区在底层技术上做一比一竞争，更现实的差异化应该来自：
  - 面向普通用户的产品化包装
  - 本地与云端的一键部署
  - 更好的安全、权限可视化和修复能力
  - 更清晰的场景表达与冷启动引导
  - 与硬件、录音、知识库的更深结合
- 从战略上看，真正稀缺的不是再做一个 Agent 外壳，而是打通 `context 积累 -> 记忆沉淀 -> Agent 执行 -> 市场生态 -> 商业化` 的连续链路。

# AI 手机 / 通用 Agent 手机卖点研究：软件与硬件拆解

- 日期：2026-03-26
- 作者：Codex
- 状态：Draft v1

## 一句话结论

如果把这条赛道看清楚，今天市场上真正成立的不是“完全替代 app 的通用 agent 手机”，而是：

> **把手机从“装很多 AI 功能的手机”推进到“一个可感知、可调用、可执行、可随身进入真实场景的 AI 入口”。**

也就是说，当前主流卖点不是纯硬件，也不是纯模型，而是：

- 软件上卖 `更自然的入口 + 更强的上下文 + 更少切 app + 更像真人助理`
- 硬件上卖 `更强的端侧算力 + 更好的传感器 + 更长续航 + 更可信的隐私安全`

但一个关键判断也很明确：

> **截至 2026 年 3 月，这个市场绝大多数产品仍然是 “AI feature phone”，不是成熟的 “general agent phone”。**

## 1. 我先给分类判断

现在所谓 `AI 手机 / Agent 手机`，实际上已经分成三类。

### 1.1 第一类：AI feature phone

代表：

- Apple iPhone 16 / 16e / 16 Pro 系列
- Samsung Galaxy S25 系列
- OPPO Find / Reno 新一代 AI 手机
- Nothing Phone 3a / 3a Pro

这类产品的特点是：

- 手机本体仍然是传统智能手机
- 核心卖点还是手机基本盘：相机、续航、性能、系统
- AI 是一层新的 OS 能力和交互层

它们今天最成功，因为：

- 不改变用户已有习惯
- 不要求用户放弃 app 生态
- AI 卖点可以叠加到已有硬件升级节奏上

### 1.2 第二类：AI-first / assistant-first phone

代表：

- Deutsche Telekom + Perplexity 的 `AI Phone / T Phone 3`
- HONOR 正在推进的 `intelligent phone / mobile AI agent` 方向

这类产品的特点是：

- 不只是“手机里有 AI”
- 而是想把 AI assistant 升成更主的入口
- 试图让用户少切 app，更直接通过语音、按钮、相机和锁屏触发 agent

它们更接近“通用 agent 手机”的早期形态。

### 1.3 第三类：独立 agent 硬件

代表：

- rabbit r1
- Humane Ai Pin

这类产品更激进，想跳过传统手机形态，直接做“AI companion device”。

但现实也很清楚：

- rabbit 还在持续 OTA 重做软件体验
- Humane 的消费者 Ai Pin 业务在 2025 年 2 月停止

这说明：

> **用户想要的是 agent 能力，不一定想为 agent 单独再带一个设备。**

所以至少在这一阶段，手机仍然是最现实的 agent 宿主。

## 2. 今天这条赛道真正的软件卖点是什么

如果只看对用户真正有感知的卖点，我认为软件层已经稳定收敛到 5 个方向。

### 2.1 卖点 1：从“输入问题”变成“让手机看见上下文”

这是 AI 手机和上一代语音助手最大的区别。

现在主流产品都在强调：

- 相机输入
- 屏幕共享
- 图片理解
- 实时视觉对话

典型例子：

- Samsung 在 2025 年 4 月 7 日宣布 Galaxy S25 系列上线 Gemini Live 的实时视觉对话能力，用户可以按住侧边键，让 AI 直接看见你正在看的东西并实时对话。
- Android 官方在 2025 年 4 月 14 日介绍 Gemini Live 的 camera / screen sharing，明确把手机摄像头和屏幕变成 AI 对话上下文。
- Apple 在 2024 年 10 月 28 日发布 Apple Intelligence 可用时，强调 iPhone 16 的 `Camera Control + visual intelligence`，让手机不只是“问答”，而是“看见物体和地点并与信息互动”。

这类卖点为什么重要：

- 因为手机是最强的随身传感器
- 相比 PC，手机更适合实时输入真实世界
- 一旦 AI 能“看见你所见”，它就更像 agent，而不是聊天框

### 2.2 卖点 2：从“回答问题”变成“帮你做事”

这一层是真正的 agent 感。

典型表达包括：

- 建日历
- 写邮件
- 打电话
- 发消息
- 导航
- 预订餐厅或打车
- 跨 app 执行动作

最典型的是 Deutsche Telekom 的 AI Phone。

Telekom 在 2025 年 3 月 3 日和 2025 年 8 月 14 日两次对外强调：

- 这台手机的核心卖点是 Perplexity Assistant
- 可以通过锁屏按钮或双击电源键触发
- 不只回答问题，还能做邮件、日历、通话、导航、预订和购物相关动作
- 官方甚至直接把卖点说成“不用在 app 之间来回切换”

HONOR 在 2025 年 3 月 2 日公布 ALPHA PLAN 时，也明确把未来方向定义成 `GUI-based personal mobile AI agent`。

这说明：

> **AI 手机真正要卖的，不是“更聪明的搜索框”，而是“一个比 app 更上层的行动入口”。**

### 2.3 卖点 3：从“单次对话”变成“长期个性化助理”

这一层是从 assistant 升级到 companion 的关键。

Samsung 的表达最完整：

- Galaxy S25 系列把 `Personal Data Engine` 作为核心卖点
- 它会在设备上组合并处理个人数据，给出个性化建议
- 数据由 Knox Vault 保护，并允许部分能力在端侧或云端之间选择

Nothing 的 `Essential Space` 是另一种更轻的路径：

- 用户通过 `Essential Key` 快速抓取截图、语音记录和内容
- 再由系统做智能分析、摘要和信息提取
- 本质上是在做“第二记忆”或“个人轻量记忆层”

这说明当前市场已经默认接受一个方向：

> **AI 手机要有记忆，不一定是无限长期记忆，但一定要能利用个人上下文。**

### 2.4 卖点 4：更低摩擦的 AI 入口

AI 手机一个非常真实的竞争点，不是模型多强，而是：

> **调起 AI 的成本有多低。**

现在主流产品都在做物理或系统级入口：

- Samsung：按住侧边键
- Apple：Camera Control、Action button、锁屏入口
- Telekom：锁屏 `Magenta AI` 按钮或双击电源键
- Nothing：`Essential Key`

Nothing 的做法尤其值得注意：

- `Essential Key` 不只是快捷键
- 它把 AI 卖点硬件化、肌肉记忆化
- 双击进 Essential Space，短按截图，长按直接录音

这说明：

> **AI 手机上的一个核心卖点，不是“能不能做”，而是“能不能在 1 秒钟内被调起”。**

### 2.5 卖点 5：可信任的隐私和安全

一旦手机开始接管：

- 相机
- 通讯
- 日历
- 位置
- 文件
- 屏幕内容

用户会天然担心隐私。

所以今天主流 AI 手机的软件卖点里，安全已经变成主叙事的一部分。

最明显的是：

- Apple 把 Apple Intelligence 的核心叙事之一直接定义为 privacy
- Samsung 把 `Personal Data Engine + Knox Vault` 放到 Galaxy S25 的主卖点区域
- OPPO 在 2025 年 3 月 3 日公开强调 `Private Computing Cloud`
- Android 在 Gemini Live 的 camera / screen sharing 说明里，也把数据默认不用于产品改进作为明确隐私控制点

所以：

> **通用 agent 手机如果没有可信的隐私故事，卖点会天然打折。**

## 3. 今天这条赛道真正的硬件卖点是什么

如果软件层卖的是“更像助理”，硬件层卖的就是“让这个助理真的随时可用”。

我认为硬件卖点主要集中在 6 个方向。

### 3.1 卖点 1：端侧 AI 算力

这是 AI 手机的第一层硬件底座。

典型代表：

- Snapdragon 8 Elite
- MediaTek Dimensity 9400 / 9400+
- Apple A18 / A18 Pro

Qualcomm 在 Snapdragon 8 Elite 产品资料里强调：

- 支持更长 token 输入
- Sensing Hub 理解个人上下文
- Hexagon NPU 更快且功耗效率更高
- 多模态 Gen AI 可直接在手机上完成

MediaTek 在 2025 年 4 月 10 日发布 Dimensity 9400+ 时，直接把 `agentic AI capabilities` 写进 SoC 卖点。

这意味着：

> **AI 手机的硬件卖点，不只是跑分，而是“能否在设备上稳定跑更复杂的多模态与 agent 流程”。**

### 3.2 卖点 2：续航、散热、持续可用性

通用 agent 手机不是一次性拍照设备，而是潜在的：

- 常开语音入口
- 相机输入入口
- 后台上下文感知入口
- 高频摘要和执行入口

这会天然抬高：

- 电池压力
- 热管理压力
- 长时间稳定性要求

所以 AI 手机的硬件卖点必须回到：

- 大电池
- 更高能效比
- 稳定散热
- 更低功耗的 NPU / sensing hub

Telekom 的 T Phone 3 虽然不是高端机，但也把 `5000mAh 电池`、`25W 快充`、升级后的芯片组写成 AI Phone 的重要卖点。

### 3.3 卖点 3：相机、麦克风和显示器变成 agent 传感器

这点非常重要。

在 AI 手机时代，相机不再只是拍照模组，而是：

- 视觉输入设备
- 场景理解入口
- 实时世界感知界面

麦克风也不只是打电话，而是：

- 长语音输入
- 实时翻译
- 环境理解
- 语音控制入口

显示和折叠形态也会影响 agent 的使用方式：

- Fold 大屏适合屏幕共享、边看边聊、跨 app 协同
- Flip 的折叠形态甚至能把手机变成“自带支架的视觉 agent”

Google/Android 在 2025 年 7 月面向 Galaxy Z 系列的官方文章里，已经开始直接把 `screen share + camera share + Flex Mode` 当成 AI 场景卖点。

### 3.4 卖点 4：物理按钮和触发件

这属于被低估但很关键的硬件卖点。

Apple 的 `Camera Control`
Nothing 的 `Essential Key`
Telekom 的 `Magenta AI` 锁屏按钮

这些设计背后的逻辑一致：

- 通用 agent 必须有低摩擦唤起
- 这个入口如果只藏在 app 里，频次上不去
- 一旦有物理触发件，就能把 AI 变成接近相机快门那样的行为习惯

所以：

> **AI 手机的硬件卖点里，新增“AI 专用物理入口”会越来越常见。**

### 3.5 卖点 5：连接能力

如果手机真要承担 agent 角色，它必须比传统手机更强调：

- 5G
- Wi‑Fi 7
- 更强定位
- 更强蓝牙和近距连接
- 卫星能力

原因很简单：

- agent 很多时候要在线
- 要跨 app、跨设备、跨服务工作
- 要在移动环境下保持响应

Qualcomm 和 MediaTek 在各自新一代平台里都把连接能力和 AI 一起讲，而不是单独讲。

这背后的逻辑是：

> **agent 手机不是“本地模型机器”，而是“边缘设备 + 云服务 + 外部生态”的连接终端。**

### 3.6 卖点 6：安全硬件

如果 agent 要读取：

- 个人上下文
- 私密数据
- 屏幕内容
- 通信内容

那么安全硬件就从“锦上添花”变成“产品成立条件”。

Samsung 的 `Knox Vault`
Apple 的安全和隐私体系
各家开始强调的本地加密与可信执行

这些不只是企业功能，而是 AI 手机的底层卖点。

## 4. 从真实市场看，什么卖点最容易打动用户

如果回到用户而不是技术，我认为今天最能成交的卖点并不是“AGI 在你口袋里”，而是下面几类。

### 4.1 高频刚需卖点

- 实时翻译
- 屏幕和相机理解
- 写邮件 / 总结 / 日历 / 待办
- 搜索和推荐更快
- 图片和视频智能编辑

Deutsche Telekom 在 2025 年 8 月 14 日的对外话术里提到，Kantar 的欧洲消费者调研里，最想要的 AI 功能包括：

- 实时翻译
- 相机增强
- Circle to Search 类能力

这和现在主流手机厂商的卖点是高度一致的。

### 4.2 比“模型更强”更有感知的卖点

普通用户不太会因为：

- 参数更多
- 上下文更长
- benchmark 更高

而买手机。

他们更可能因为：

- 一键调起
- 真的省步骤
- 少切 app
- 相机更好用
- 语音更自然
- 能直接做事

这意味着在消费市场里：

> **agent 手机的卖点必须被翻译成更短路径、更少步骤和更强随身性。**

## 5. 这条赛道里，哪些路径看起来更对，哪些路径已经暴露问题

### 5.1 看起来更对的路径

**路径 A：在成熟手机里把 AI 逐渐抬升为 OS 层入口**

代表：

- Apple
- Samsung
- OPPO
- HONOR

这条路径更稳，因为：

- 不要求用户抛弃手机主机位
- 不破坏 app 生态
- 可以把 AI 逐步系统化

**路径 B：用一个具体按钮或入口把 AI 高频化**

代表：

- Camera Control
- Essential Key
- 侧边键调起 Gemini Live
- 锁屏 AI 按钮

这是非常真实的产品动作，因为它解决的是触发成本。

**路径 C：从“问答”升级到“跨 app 执行”**

这才是通用 agent 手机真正的方向。

如果只能回答问题，它只是 assistant。
如果能跨 app 帮你做事，它才开始接近 agent。

### 5.2 已经暴露问题的路径

**路径 D：只卖“app-free”叙事**

rabbit r1 的故事最典型。

它最初最强的叙事就是：

- app-free
- 自然语言 OS
- LAM 代替 app

这个方向很有想象力，但现实问题也暴露得很快：

- 交付体验不如预期
- 用户对“第二设备”的容忍度有限
- 软件体验比故事本身更难补

**路径 E：把 agent 单独做成一块硬件，但没有稳定生态**

Humane Ai Pin 的消费者业务在 2025 年 2 月终止，基本说明了一件事：

> **如果 agent 设备既没有成熟生态，也没有比手机强很多的使用频率和必要性，它很难成立。**

## 6. 我对“通用 Agent 手机”这个方向的最终判断

### 6.1 现在最真实的判断

截至 `2026-03-26`，我认为：

- `AI 手机` 已经是成立品类
- `通用 Agent 手机` 还处于早期定义阶段

更直白一点说：

- 市场已经接受“手机里必须有 AI”
- 但还没有完全接受“手机会成为真正跨 app 的通用 agent”

今天大多数品牌卖的还是：

- AI 搜索
- AI 相机
- AI 摘要
- AI 写作
- AI 翻译
- AI 入口

而不是完整意义上的：

- 全局任务代理
- 统一个人行动系统
- 稳定跨 app 编排层

### 6.2 未来 12-24 个月最值得关注的卖点升级

我会重点看 4 个方向。

**方向 1：跨 app action 的稳定性**

谁能真正让用户说一句话，就稳定完成：

- 日历
- 通信
- 导航
- 购买
- 预订
- 文件处理

谁就更接近真正的通用 agent 手机。

**方向 2：个性化记忆层**

不是聊天记忆，而是：

- 用户偏好
- 日程
- 历史行为
- 常用工作流
- 多源内容收纳与提取

这会决定手机是“工具”，还是“长期助理”。

**方向 3：物理入口**

我判断未来会有更多：

- AI 键
- 相机 / 语音 / 锁屏专用入口
- 可穿戴联动入口

这会极大影响 AI 使用频次。

**方向 4：隐私与安全架构**

一旦 agent 真正开始读取屏幕、邮件、日历和位置，用户最终会用脚投票给：

- 延迟更低
- 更可信
- 权限更透明
- 更可控

的方案。

## 7. 卖点层的产品判断

我会写成下面三句话。

### 7.1 第一条

> **AI 手机今天最强的卖点不是“最强模型”，而是“最短路径地理解你当前情境，并帮你完成下一步动作”。**

### 7.2 第二条

> **通用 Agent 手机短期不会先以“颠覆手机形态”胜出，而更可能先在成熟手机形态里，通过 OS、按钮、相机和跨 app 执行逐步长出来。**

### 7.3 第三条

> **软件决定 agent 感，硬件决定 agent 能否高频、可信、持续地被使用。**

## 8. 如果你是强 AI 平台公司，这条线要看的不是单个产品机会，而是“手机会不会成为 AI 时代的主终端”

这一节直接按平台公司视角来看，默认你已经具备：

- 自有模型与推理体系
- OS 控制权
- 手机硬件团队
- 端云协同能力
- 生态和分发能力

在这个前提下，问题就不再是“能不能切进来”，而是：

> **如果模型、OS 和设备都在你手里，AI 手机 / 通用 agent 手机是不是下一代个人计算平台的主战场。**

我的判断是：是，而且这件事的重要性高于“再做一个独立 app”，也高于“做一个会聊天的系统功能”。

### 8.1 为什么这会是平台型强公司的主战场

因为手机不是普通硬件，而是今天个人计算里最密集的默认入口集合。

- 它离用户最近，触达频次最高
- 它天然掌握相机、麦克风、位置、通知、锁屏、通讯和支付这些高价值原生入口
- 它是个人身份、关系、日程、内容和行为最连续的设备
- 如果 agent 要从“回答问题”变成“长期代理”，它迟早要占住用户每天几十到上百次的微交互节点

而这些节点，最集中地就发生在手机上。

所以如果你同时拥有模型、OS 和设备，战略目标就不该只是：

- 给手机加更多 AI feature

而应该是：

- 把手机从“装了 AI 的手机”推进成“AI 时代默认的个人操作入口”

> **PC 更像复杂工作台，手机更像随身默认入口；真正的平台价值，在于把默认入口变成默认 agent。**

### 8.2 这个视角下，最容易看错的三件事

第一，不要把胜负手理解成“模型参数谁更强”。

用户对手机 agent 最直接的感知，通常不是 benchmark，而是：

- 调起摩擦够不够低
- 上下文是否连续
- 动作能不能真的完成
- 结果是否可信、稳定、可追踪

第二，不要把胜负手理解成“先造一台 AI 手机新硬件”。

如果硬件没有和 OS 入口、agent runtime、记忆层、权限系统打通，它就很容易退化成：

- 一台多了几个 AI 按钮的手机
- 一个带 AI 卖点的硬件 SKU

这不是真正的平台壁垒。

第三，不要把目标理解成“立刻替代所有 app”。

成熟路径更可能是：

- 先拿入口
- 再拿记忆和上下文
- 再拿动作分发权
- 最后让 app 逐渐退到被调度的工具层

> **平台型公司的优势，不在于把所有 app 都自己做掉，而在于把 app 之上的调度权、入口权和记忆权拿到手。**

### 8.3 真正该回答的战略问题是什么

如果换成平台视角，我认为真正该回答的不是“要不要做 AI 手机”，而是下面四个问题：

1. 你能不能把 agent 做成 OS 的默认入口，而不是一个可有可无的 feature。
2. 你能不能围绕几个高频场景，把 `capture -> understand -> act` 做成日常习惯，而不是一次性 demo。
3. 你能不能把第三方 app 和服务接进来，让它们变成 agent 可调度的动作层。
4. 你能不能把本地、云端、权限、安全和硬件协同成一个可信 runtime。

如果这四件事做成，手机就不只是“AI 能力的承载物”，而会变成：

- 个人默认的 agent 终端
- 组织流程的第一现场入口
- 下一代应用生态的调度中心

## 9. 用户价值最密集的地方在哪里

如果只看用户价值，而不先看技术酷不酷，我认为最密集的价值集中在四类场景。

### 9.1 场景 1：看到什么，就立刻记住并形成下一步动作

这是今天最真实、也最自然的手机 agent 场景。

典型输入：

- 截图
- 相机拍到的内容
- 语音备忘
- 聊天里的链接
- 屏幕上的商品、地点、联系人、任务信息

用户真正想要的不是：

- “给我解释一下这是什么”

而是：

- 帮我记住它
- 帮我整理它
- 帮我把它变成稍后会被执行的动作

最典型的用户价值是：

- 少忘事
- 少丢上下文
- 少回头翻相册和截图
- 少做二次录入

如果把它展开到日常生活里，典型例子其实很多：

- 在社交媒体或聊天里看到一家想去的餐厅，截图后自动识别店名、位置，并加入“想去清单”
- 朋友发来一个地址、航班号或演出票信息，手机自动抽取并生成导航、日历提醒或同行待办
- 临时拍下停车位号码、快递柜取件码、门禁密码、会议室号，系统自动记住并在正确时间弹出
- 看到一张优惠券、缴费通知、续费页面或报名截止时间，自动识别 deadline 并提醒
- 刷到一个商品、清单或食谱，截图后自动进购物清单或“以后再买”列表

这一类场景为什么密集：

- 它每天都发生
- 它高度依赖手机
- 它天然适合相机、截图、按钮、语音这些输入方式

Nothing 的 `Essential Key + Essential Space` 本质上就是在验证这件事：

> **手机上有大量“稍纵即逝的价值瞬间”，用户真正缺的是把这些瞬间变成可行动记忆。**

如果从平台视角看，这是一条值得重视的消费级 / prosumer 价值带。

但它也有明显代价：

- 付费逻辑未必天然强
- 很容易被手机厂商自己吸收
- iOS 上平台约束更重

所以它更适合：

- Android-first
- 创作者 / 知识工作者 / founder / 高信息密度用户
- 从 `capture -> structure -> action` 这条链切

### 9.2 场景 2：出门状态下的“生活事务执行”

这是很多厂商正在讲的场景：

- 订餐
- 导航
- 发消息
- 加日历
- 购物
- 出行
- 翻译

从用户价值看，它非常直观：

- 少切 app
- 少打字
- 少步骤
- 手忙脚乱时更方便

如果把这一类能力翻译成更生活化的提效场景，常见会是：

- 出门路上说一句“帮我顺路把干洗店、取快递和买猫粮排一下”，系统直接规划路线和提醒顺序
- 在商场或机场里看到停车位、登机口、柜台信息，手机自动记住并在需要时快速调出
- 看完医生、牙医或体检报告后，自动生成复诊提醒、购药清单和家人同步消息
- 家庭协同场景里，把“接娃、买菜、缴费、维修预约、周末出行”这类碎事务整理成共享待办
- 遇到外语菜单、路牌、快递说明、合同重点，边看边翻译，再直接形成下一步动作而不是只停留在理解
- 收到航班延误、快递派送、物业通知、缴费短信后，自动转成提醒、改期建议或沟通草稿

#### 9.2.1 生活场景里，哪些适合自己做，哪些不该自己下场

如果把这条线再往产品边界上拆，我会把它分成两层。

**更适合自己做成“生活提效层”的：**

- 记账：拍小票、识别账单、自动归类支出、月度总结、异常提醒
- 记忆与提醒：停车位、取件码、门禁密码、复诊时间、续费日、缴费日
- 家庭协同：共享购物清单、接娃安排、家庭待办、老人用药提醒
- 路径与顺序优化：把多个碎任务按位置、时间和优先级自动排顺路
- 信息抽取：从短信、截图、聊天里提取地址、时间、联系人、票号、订单号
- 个人轻决策：该不该续费、这个月花销在哪、哪些待办最容易忘、今天顺路能做掉什么

这类能力的共同点是：

- 不依赖重供给侧
- 不需要自己经营商家网络
- 主要价值在于 `capture -> organize -> remind -> act`
- 更像系统层和 agent 层应该提供的通用效率能力

**更不适合自己下场做成“本地生活巨头业务”的：**

- 点外卖
- 订电影票
- 打车和即时配送撮合
- 酒店 / 机票 / 门票交易平台
- 到店团购和优惠供给网络
- 餐厅预订、商家履约、售后客服这类重交易链路

不是说这些动作不该被 agent 触达，而是：

- 更适合通过生态连接和动作分发完成
- 不适合自己重做一遍商家、供给、履约和支付网络

所以更合理的边界是：

> **自己做“生活提效层”和“默认入口层”，把重交易本地生活留给第三方生态。**

但从平台战略看，这更适合作为系统级标配能力，而不是唯一差异化来源。

原因是：

- 这是 Apple、Google、Samsung、Perplexity 们最先会抢的入口
- 用户的“默认助理心智”天然会集中到系统级玩家
- 单靠这条线，很难形成足够深的长期护城河

也就是说：

- 这是大厂主战场
- 也是系统品牌必须覆盖的基本盘

### 9.3 场景 3：手机作为“个人工作终端”的移动提效

这是我觉得最值得从平台扩张角度认真看的方向。

因为在很多岗位里，手机不是辅助设备，而是主设备。

这一节更偏：

- 个人或单岗位视角
- “让一线人员自己更省时间、更少补录、更少出错”
- 就算还没深度回写组织系统，单人效率提升也已经成立

典型岗位包括：

- 外勤销售
- 到店拜访和渠道督导
- 现场服务 / 安装 / 运维
- 保险查勘
- 门店店长 / 区域经理
- 物流与配送现场管理
- 物业、工程、巡检等现场工作

这些岗位的共同特点是：

- 工作发生在现场
- 现场有大量相机、语音、位置和即时沟通
- 事后补录、写总结、回 CRM / 工单系统的成本很高
- 手机就是第一工作终端

用户真实价值非常明确：

- 当场采集信息，而不是回去补
- 拍完照、说完话就自动形成结构化记录
- 自动生成下一步动作
- 需要时再进入 CRM / 工单 / 表单 / 日报
- 减少“现场做事，晚上补文书”的痛苦

这类场景为什么特别重要：

- workflow 明确
- ROI 明确
- 手机原生能力真有优势
- 就算不先吃掉组织系统，也能先把个人工作终端效率做出来

如果用平台语言来讲，这类方向的优势就在于：

> **它不是先讲一个抽象的 AI 愿景，而是先把一个很痛、很重复、很移动端原生的工作流彻底吃下来。**

这是我对这条赛道最偏好的价值放大场景之一。

### 9.4 场景 4：围绕“人和关系”的多人轻协同

这个方向我现在认为比“手机进入 B 端组织系统”更成立，因为手机首先是：

- 通讯设备
- 关系设备
- 协同设备

如果说 `9.3` 的核心是“我自己把工作做快”，那 `9.4` 的核心就是：

- 我怎么更顺地和别人协同
- 我怎么少漏回复、少漏安排、少漏跟进
- 我怎么把分散在聊天、截图、语音里的协同信息整理成共享动作

也就是说，这里不一定要先碰重型组织系统，而是更像：

- 聊天协调层
- 家庭协同层
- 朋友 / 同伴 / 小团队的共享行动层

典型场景不是：

- 让用户和 AI 聊天

而是：

- 手机把分散的人际协同信息接住
- agent 把信息结构化
- 再推进到共享待办、日历、提醒、消息草稿和轻量协同动作

比如：

- 家庭群里零散出现“接娃、买菜、缴费、周末安排”，自动整理成共享待办和时间表
- 朋友约饭或出行时，聊天里的地点、时间、人员分工自动抽成日历、导航和提醒
- 对话里提到“记得回某人消息、催一下合同、给家里买药”，自动形成跟进列表
- 旅行过程中，航班、酒店、集合地点、同行人消息自动归成一个行程卡片
- 你口述一段要发给同事、家人或合作方的安排，系统先整理成清晰消息草稿再发

这条线的用户价值非常清楚：

- 降低人与人之间的协调摩擦
- 减少忘回复、忘安排、忘同步
- 让聊天内容更容易进入共享行动，而不是淹没在消息流里

所以两者的差别可以直接压缩成一句话：

> **`9.3` 更像“让我自己更高效”，`9.4` 更像“让我和别人协同得更顺”。**

如果你让我在“通用 agent 手机”这个大方向里选一个最像平台级价值放大的表达，我会更倾向于：

> **移动端关系协同 agent**

因为这条逻辑能把：

- 设备入口
- 模型理解
- 通讯与关系上下文
- 共享动作生成

连成一条真正有复利的链。

### 9.5 按语音链路 / 视频链路 / 文字链路重看高密度场景

这个视角是对的，因为移动端很多看似不同的场景，底层其实是同一种结构：

> **capture -> understand -> structure -> action -> sync**

区别主要不在“是否是 agent”，而在：

- 第一输入模态是什么
- 哪一步必须本地做
- 哪一步必须云端做
- 最后落到哪个系统里

#### 9.5.1 语音链路

这条链在移动端尤其强，因为很多场景里：

- 手不方便打字
- 情境稍纵即逝
- 说话比写字快
- 现场信息带有大量口头上下文

所以很多移动 workflow 本质上都是：

> **录音 -> 转写 -> 总结 -> 代办 -> 回写系统**

#### 典型场景

- 销售拜访后口述客户纪要
- 巡检 / 现场服务结束后的问题记录
- 门店巡店后的语音点评
- founder / 投资人 / 创作者的灵感和待办 capture
- 客服复盘或服务人员的现场说明
- 开车或走路时随口记下“待买清单、待办、要回复的人和要问医生的问题”
- 带娃、照护老人或处理家庭事务时，随口记录一串琐事并自动拆成提醒和家庭协同任务

#### 语音链路的标准流程

1. 语音触发
- 按键
- 锁屏
- 耳机
- 语音助手

2. 本地或近端转写
- 先把声音稳定变成文字

3. 结构化理解
- 谁
- 什么问题
- 下一步动作
- deadline
- 相关客户 / 项目 / 门店 / 工单

4. 动作生成
- 代办
- 日历
- CRM 跟进
- 巡检问题单
- 会议纪要

5. 系统回写
- CRM
- 工单系统
- Notion / 飞书 / Jira / Linear

#### 语音链路的用户价值

- 最低输入摩擦
- 最适合移动和现场状态
- 最容易保留第一现场的主观判断和语义细节

所以你说得对：

> **record 工作流里，有很大一类其实都是语音为主。**

它们表面上不同，但底层都是“说出来 -> 结构化 -> 推进行动”。

#### 9.5.2 视频链路

这里的“视频链路”不一定非得是长视频，也包括：

- 相机实时画面
- 连续拍摄
- 视频片段
- 带时间和场景上下文的视觉采集

这条链最适合那些：

- 视觉证据比口头证据更关键
- 现场环境复杂
- 需要 OCR、识别、定位、比对

#### 典型场景

- 巡检和工程现场问题识别
- 保险查勘
- 门店陈列和货架审计
- 设备故障记录
- 收据 / 单据 / 票据采集
- 现场安装或维修的前后对比记录
- 冰箱、药盒、收纳柜或购物小票拍一下，自动识别缺货、临期和补货清单
- 停车位、快递柜、酒店房号、展会展位、景区导览图拍一下，后续需要时直接调出

#### 视频链路的标准流程

1. 相机 / 视频 capture

2. 本地预处理
- OCR
- 目标检测
- 关键帧抽取
- 位置和时间拼接

3. 云端或重模型理解
- 问题归因
- 场景判断
- 与模板或标准对比

4. 结构化输出
- 问题列表
- 风险等级
- 整改建议
- 需要的物料 / 工具 / 下次动作

5. 系统回写
- 工单
- 质检记录
- 巡检报告
- 项目管理系统

#### 视频链路的用户价值

- 证据更客观
- 降低文字描述歧义
- 特别适合“先看图，再做判断”的行业场景

#### 9.5.3 文字链路

这条链最适合那些原本就已经以数字文本形式存在的场景。

输入可以是：

- 截图
- 聊天记录
- 邮件
- 文档片段
- 页面内容
- 表单和审批记录

很多人低估了这一类，但其实手机上也很高频，因为：

- 用户天天在看消息
- 天天在看截图
- 天天在复制粘贴信息

#### 典型场景

- 客户聊天截图变 follow-up
- 邮件和 IM 内容变任务
- 订单、地址、联系人信息自动抽取
- 审批结论和需求反馈自动归纳
- support chat 转成 ticket 或 backlog
- 短信里的快递状态、航班变更、账单到期、医院预约自动转成提醒和待办
- 家庭群、朋友群里的聚餐地址、时间、分工安排自动抽取成日历、导航和共享清单

#### 文字链路的标准流程

1. 文本 capture
- 复制
- 转发
- 截图
- 分享到 agent

2. 本地或云端 OCR / NLP

3. 结构化
- 人
- 地点
- 时间
- 问题
- 下一步动作

4. 输出
- 总结
- 代办
- 日历
- CRM / 工单 / 项目系统同步

#### 文字链路的用户价值

- 对已有数字内容改造成本最低
- 不需要额外拍照或重新口述
- 最容易变成“分享给 agent”式入口

#### 9.5.4 很多移动工作流不是单一链路，而是“语音为主，视频和文字补全”

这一点对产品设计尤其重要。

很多真实移动 workflow 不会只依赖一种模态。

更常见的结构其实是：

- **语音负责快速 capture**
- **视频负责补现场证据**
- **文字负责进入正式记录、共享清单或合适系统**

比如外勤销售场景：

- 先语音说三分钟拜访纪要
- 再拍一下门店或合同材料
- 最后自动生成 CRM 记录和待办

比如巡检场景：

- 先拍现场
- 再语音解释问题
- 最后结构化生成工单

比如生活事务场景：

- 先截图朋友发来的餐厅和地址
- 再语音补一句“下周三晚上提醒我约这个地方”
- 最后自动生成日历、导航和待办

再比如家庭协同场景：

- 先拍药盒或购物清单
- 再语音说“这周末顺路补齐，缺了牙膏和儿童退烧药”
- 最后同步成家庭共享待办

所以如果从产品和平台设计角度看，不一定要追求“全模态全能”，而更应该先回答：

> **这个 workflow 的主输入模态到底是什么？**

因为这会决定：

- 入口怎么设计
- 本地预处理怎么设计
- 云端推理怎么设计
- 用户为什么愿意高频使用

#### 9.5.5 如果把这个视角浓缩成一句话

我会这么写：

> **移动端很多 agent 场景表面上是“通用助理”，底层其实是不同输入链路驱动的 workflow 自动化；而其中最强的一类，往往是以语音为第一入口，再由视频和文字补全证据，最后进入提醒、日历、共享清单或其他合适系统。**

### 9.6 最容易打动用户的 10 个瞬间

如果把前面的消费、生活、工作和组织场景都拉平来看，我认为最容易让用户当场感到“这个 agent 真的有用”的瞬间，通常是下面 10 类。

1. 朋友发来一家餐厅和地址，你截个图，手机自动存成“想去清单”，并在约饭时给出导航和时间建议。
2. 你拍下停车位号码、快递柜取件码或酒店房号，几个小时后系统在你真正需要时直接把它调出来。
3. 你收到航班变更、账单到期、医院预约或物业通知短信，手机不只是提醒你看，而是直接生成待办、时间和下一步动作。
4. 你在外面看到外语菜单、路牌、说明书或合同重点，手机边翻译边抽取关键信息，再形成可执行动作。
5. 你走路、开车或洗澡前突然想到一串事，只说十几秒，系统就自动拆成购物清单、提醒、要回复的人和日历事项。
6. 你拍一张购物小票、发票或账单截图，系统自动记账、归类，并提示这个月哪些支出异常。
7. 你看完体检报告、处方或药盒，手机自动整理出复诊时间、补药提醒和需要同步给家人的信息。
8. 销售拜访结束后，你边走边说三分钟，系统自动生成客户纪要、follow-up 和 CRM 更新草稿。
9. 巡检、维修或查勘现场，你拍照再补一段语音，系统自动归类问题、生成工单和整改建议。
10. 家庭群或朋友群里零散出现时间、地点、分工和采购事项，系统自动抽成共享清单、日历和顺路计划。

这 10 个瞬间的共同点是：

- 输入摩擦极低
- 用户马上能感知节省的步骤
- 不需要先学习复杂玩法
- 很容易形成“以后我就默认这样用”的习惯

## 10. 如果从平台战略给这条赛道排优先级

### 10.1 第一优先：把 agent 做成 OS 的默认入口和默认分发层

这件事是根。

如果 agent 不能稳定占住：

- 锁屏
- 物理按钮
- 相机
- 分享面板
- 系统搜索
- 通知和提醒

它就很容易退化成：

- 一个需要被主动打开的 app
- 一个演示时很强、日常使用很弱的 AI 能力

所以第一优先不是“堆更多功能”，而是：

- 统一入口
- 统一记忆
- 统一权限
- 统一动作分发

也就是先把 agent 做成 OS 的默认 runtime，再谈后面的场景扩张。

### 10.2 第二优先：围绕高频场景，把 `capture -> understand -> act` 主链做深

平台能力不能只停在“系统里有个 AI”。

必须有几条高频链路，能让用户和组织明确感受到：

- 比以前少步骤
- 比以前少切 app
- 比以前少丢上下文
- 比以前更快形成可执行动作

我认为最值得优先打深的三类链路是：

- 个人记忆和 capture-to-action
- 手机主工作终端岗位的移动 workflow
- 组织里的移动证据到执行

这一步的意义，不只是做几个好用场景，而是建立：

- 高频使用习惯
- 场景级数据闭环
- 模型和 agent 策略的持续优化空间

### 10.3 第三优先：把 app 生态做成 agent-ready，而不是自己包办所有服务

如果你真把手机看成下一代 agent 平台，最终一定要面对一个问题：

> **是你自己做完所有服务，还是让 app 生态都变成你可调度的动作层。**

我更倾向于后者。

原因是：

- 用户不会放弃成熟 app 生态
- 不同行业的深能力仍然在 app 和 SaaS 里
- 真正的平台壁垒，不是自己提供一切，而是别人都要通过你这一层被调用

所以第三优先应该是：

- 建好 intent / tool / action 的接入层
- 建好权限、确认和审计机制
- 让第三方 app 能安全接入 agent 分发体系

这样 app 不会立刻消失，但会越来越像 agent 身后的工具层。

### 10.4 第四优先：再用硬件把体验差距放大，而不是用硬件替代产品战略

如果你已经有手机团队，硬件当然重要。

但硬件最强的价值，不是单独成立，而是把上面三件事放大。

真正值得投入的硬件方向是：

- 更稳定的端侧 AI 算力
- 更强的续航和热管理
- 更适合 agent 的相机 / 麦克风 / 传感器组合
- 更低摩擦的物理触发件
- 更可信的安全芯片和本地隐私计算

也就是说，硬件不是平台战略的替代品，而是：

- 频次放大器
- 可信度放大器
- 可用性放大器

## 11. 平台层的最终判断

如果把这份研究压缩成平台层的四句话，我会给出下面的判断。

### 11.1 第一条

> **对于同时拥有模型、OS 和设备能力的公司，AI 手机 / 通用 agent 手机不是边缘项目，而是下一代个人计算入口的主战场。**

### 11.2 第二条

> **真正值得争夺的，不是一台贴着 AI 标签的手机，而是“默认输入入口 + 默认个人记忆 + 默认行动分发”的系统地位。**

### 11.3 第三条

> **最能把平台能力做厚的，不是泛化聊天，而是把高频真实场景做成 `capture -> understand -> act` 的日常闭环。**

### 11.4 第四条

> **最终胜出的不会只是模型最强或硬件最激进的一家，而更可能是把模型、OS、agent、云端、本地、硬件和生态协同成一个产品系统的一家。**

## 12. 架构层补充：OS、agent、云端本地与双层协同

这一节不改前面的平台判断，只补底层架构逻辑。

如果把手机上的 `agent` 想清楚，我认为最重要的一点是：

> **在手机上，agent 不是独立于 OS 的一层，而是强依赖 OS 的入口、权限、上下文、调度和安全边界。**

这和 PC 很不一样。

在 PC 上，用户可以：

- 同时开很多窗口
- 自己决定主入口
- 用浏览器、剪贴板、文件系统把信息拼起来

但在手机上，真正决定 agent 能不能高频成立的，是 OS 级的几件事：

### 12.1 OS 控制入口

手机 agent 最重要的第一件事不是模型，而是：

- 用户怎么调起它
- 调起成本有多低
- 能不能在锁屏、侧边键、相机键、Action Button、通知、桌面小组件里出现

所以：

- Apple 的 Siri、Visual Intelligence、Shortcuts、Action Button 是系统级入口
- Samsung 的侧边键、Gemini Live 深度整合也是系统级入口
- Nothing 的 Essential Key 其实是在硬件层争夺 OS 入口

谁控制入口，谁就控制频次。

### 12.2 OS 控制权限

真正的 agent 如果要做事，就一定要碰到：

- 相机
- 麦克风
- 通讯录
- 日历
- 消息
- 邮件
- 文件
- 位置
- 屏幕内容

而这些能力在手机上不是开放平原，而是被系统严格管理的。

也就是说：

> **手机 agent 的上限，不只取决于模型能力，也取决于平台愿意把多少系统能力开放给它。**

### 12.3 OS 控制上下文

agent 真正变强，不只是因为它能推理，而是因为它能知道：

- 你现在在看什么
- 你刚刚拍了什么
- 你和谁在聊
- 你的日历里接下来有什么
- 你的位置和时间意味着什么

在手机上，这些上下文天然都先经过 OS。

所以系统层 agent 的天然优势就在于：

- 更容易组合跨 app 上下文
- 更容易在不打断用户的情况下提供下一步动作
- 更容易把 AI 变成“当前情境里的默认助手”

### 12.4 OS 控制调度

很多人谈 agent，容易只想到“回答得更聪明”。

但手机 agent 真正麻烦的其实是：

- 何时在后台跑
- 何时该提醒
- 何时该同步
- 何时该等待确认
- 何时该离线 fallback

这些事情都更像 OS 和 runtime 的问题，而不只是模型问题。

### 12.5 OS 控制信任

用户愿不愿意把：

- 个人上下文
- 屏幕内容
- 拍到的材料
- 私密记录

交给 agent，很大程度取决于：

- 它是不是系统级能力
- 权限是不是清楚
- 数据是不是本地优先
- 跨云时是否可信

这也是为什么 Apple 会把 Apple Intelligence 的叙事做成：

- on-device processing
- Private Cloud Compute
- App Intents
- Visual Intelligence

也为什么 Android 体系里，系统集成深的厂商更容易把 agent 做得“像 OS 功能”，而不是“像一个 app”。

### 12.6 所以手机上的 agent 更像一套分层结构

我更倾向于把它拆成四层：

**第一层：OS 入口层**

- 锁屏
- 物理按钮
- 相机
- 麦克风
- 通知
- 系统搜索

**第二层：系统 agent 层**

- Siri
- Gemini
- Samsung 系统层 AI
- OEM 自带 assistant

这一层负责：

- 统一入口
- 系统级上下文
- 基础跨 app 行动

**第三层：app / workflow agent 层**

- 第三方 app 里的 agent
- 某个垂直 workflow 的移动 agent

这一层负责：

- 深行业能力
- 深 workflow 逻辑
- 具体系统回写

**第四层：模型与推理层**

- 本地模型
- 云端模型
- 工具调用
- 路由和策略

这一层才是模型本身。

如果这么看，就会很清楚：

> **对平台型强公司来说，第一层和第二层决定分发权，第三层决定场景深度，第四层决定智能上限。**

### 12.7 云端和本地到底应该怎么配合

如果手机 agent 真要成立，架构上最关键的问题不是“全本地”还是“全云端”，而是：

> **什么必须放本地，什么值得上云，什么必须经过人确认。**

我现在更认可的是：

> **本地负责入口、隐私、即时性和个人上下文；云端负责重推理、长链路、多工具和更复杂的跨系统编排。**

#### 12.7.1 本地最适合做什么

本地最适合承担四类任务：

**A. 低延迟触发**

- 物理按钮触发
- 热启动
- 语音录入开始
- 相机实时预览
- 简单意图分类

**B. 敏感上下文处理**

- 本地个人记忆
- 通讯录、消息、照片、日历等高敏数据的初步读取
- 本地权限判断

**C. 轻量结构化**

- ASR 初转写
- OCR
- 截图和相机的初步解析
- 简单摘要和标签化

**D. 离线与 fallback**

- 无网时的记录
- 稳定的 capture
- 基础总结
- 待联网后同步

这也是 Apple 近两年的官方公开逻辑：

- 很多模型在设备上运行
- 再通过 Private Cloud Compute 处理更复杂请求

#### 12.7.2 云端最适合做什么

云端更适合承担：

**A. 更复杂的推理**

- 长上下文总结
- 多轮任务规划
- 多步骤工具调用
- 更复杂的多模态理解

**B. 跨系统编排**

- CRM
- 工单系统
- 企业知识库
- 多 SaaS 工具同步

**C. 组织级知识和策略**

- 团队模板
- 历史流程规则
- 组织级 KPI
- 更重的搜索和检索

**D. 持续学习和优化**

- 更复杂的召回
- 模板改进
- 质量监控

#### 12.7.3 人应该站在哪一层

很多团队会把讨论停在：

- 本地 vs 云端

但实际产品里更重要的是：

- 本地
- 云端
- 人

三者怎么分工。

因为真正影响 agent 产品体验的，不只是推理能力，而是：

- 哪些动作可以自动做
- 哪些动作需要先给人确认
- 哪些动作只能建议，不能替人执行

我更赞成这样的原则：

**本地优先处理：**

- 捕捉
- 初步理解
- 敏感信息筛选

**云端优先处理：**

- 更复杂的判断
- 更重的工作流规划
- 组织级系统同步

**人必须确认：**

- 发消息给客户
- 修改正式系统记录
- 提交审批
- 发起付款、下单、删除、发布等高风险动作

#### 12.7.4 一个更现实的架构图

如果把手机 agent 的工作链画出来，我会更偏向下面这个模型：

1. **本地 capture**
- 按钮 / 语音 / 相机 / 截图

2. **本地预处理**
- ASR / OCR / 初步分类 / 本地上下文拼接

3. **路由判断**
- 轻任务本地完成
- 重任务发云端

4. **云端推理与工具调用**
- 总结
- 结构化
- workflow planning
- 多系统回写准备

5. **人类确认**
- 对外动作
- 高风险动作
- 正式系统变更

6. **系统回写与持续跟踪**
- CRM
- 工单
- Notion
- Jira / Linear
- 日报 / 周报

#### 12.7.5 从平台架构角度，这意味着什么

这意味着真正该投入的，不是孤立地强化某一个点，而是把下面这条链打通：

- 本地 capture
- 本地预处理
- 云端理解与编排
- 人类确认
- 正式系统回写

谁把这条链设计得最顺，谁就更可能把手机做成一个可信、可持续、可规模化的 agent 终端。

### 12.8 苹果当前公开的 Siri 计划，应该怎么理解

这一节我尽量只用苹果自己的公开信号，不靠传闻。

#### 12.8.1 苹果在 2024 年 6 月给出的原始承诺

Apple 在 `2024-06-10` 介绍 Apple Intelligence 时，给 Siri 画的方向非常清楚：

- 更自然
- 更个人化
- 更懂个人上下文
- 更有屏幕感知
- 能在 app 内和跨 app 执行动作

这是第一版“新 Siri”承诺。

#### 12.8.2 2024 年 10 月和 2025 年 2 月，苹果的公开表述依然偏积极

到 `2024-10-28`，Apple 在 Apple Intelligence 首批上线时仍然说：

- Siri 会在 coming months 里获得更多能力

到 `2025-02-20`，Apple 又公开说：

- Apple Intelligence 接下来几个月会继续扩展
- 其中包括 more capabilities for Siri

也就是说，在这个阶段，苹果对外仍然是“持续推进中”的口径。

#### 12.8.3 但截至 2026 年 3 月，苹果官网上的两个关键信号依然很值得注意

截至 `2026-03-26`，苹果官方公开页面里有两个非常重要的现状信号：

**信号 1：Apple Intelligence 官网**

苹果官网的 Apple Intelligence 页面仍然写着：

- Siri 的 onscreen awareness 是 in development
- personal context 相关能力是 in development
- 这些能力会在 future software update 中提供

**信号 2：Siri for Developers 页面**

Apple Developer 的 Siri 页面同样仍然写着：

- Siri’s personal context understanding, onscreen awareness, and in-app actions are in development and will be available with a future software update

这说明一件事：

> **苹果并没有撤回“新 Siri”的方向，但截至 2026 年 3 月，最完整的那版 Siri 仍然没有被官方表述成“已经全面落地”。**

#### 12.8.4 与此同时，苹果真正已经在落地的，是 Siri 外围的能力底座

这一点特别关键。

如果你只盯着“新 Siri 什么时候 fully ship”，会漏掉苹果真正已经在做的事情。

从公开动作看，苹果更像是在先铺底座：

**A. App Intents**

- 让 app 的动作进入系统级可调用层
- 让 Siri、Spotlight、Shortcuts、Control Center 都能发现这些动作

**B. Visual Intelligence**

- 让系统级视觉理解有明确 app 接入方式

**C. Shortcuts 智能化**

- 把更复杂的个人 workflow 逐渐吸到系统 workflow 层

**D. Foundation Models framework**

- 在 WWDC25 后，开发者已经能直接访问 Apple Intelligence 的端侧基础模型
- 支持本地文本提取、总结、tool calling 等

**E. Private Cloud Compute**

- 把更复杂请求送到苹果云上，但保持其“不是传统云外包”的隐私叙事

这说明苹果的真实路线并不只是“把 Siri 做聪明”，而更像是：

> **先把 Apple Intelligence 做成 OS 基础设施，再让 Siri 成为这套基础设施最自然的对话入口。**

#### 12.8.5 我对苹果“新 Siri 计划”的当前理解

基于苹果自己的公开资料，我现在更倾向于这样的理解。

> **苹果的新 Siri 不是一个单点版本升级，而是一条分阶段路线。**

第一阶段：

- 先把 Siri 做得更自然
- 加入 ChatGPT
- 增加基础系统理解

第二阶段：

- 先把 App Intents、Visual Intelligence、Shortcuts、Foundation Models、Private Cloud Compute 这些底座做起来

第三阶段：

- 再把 personal context、onscreen awareness、in-app actions 真正稳定地汇总到 Siri 这个前台入口里

这里我要明确说一句：

> **上面这三阶段是我基于苹果官方公开信号做的推断，不是苹果官方原话。**

但我认为这个推断是合理的，因为它解释了为什么：

- 苹果承诺的 Siri 方向没有消失
- 但苹果最近更高密度公开的，其实是外围基础设施

#### 12.8.6 这对平台型强公司意味着什么

这反而是一个很重要的提醒：

> **苹果正在把“通用手机 agent 的横向能力”越来越系统化。**

这意味着，如果你也拥有模型、OS 和设备能力，就不能把战略押在一个“更会聊天的前台助手”上。

更该做的是：

- 抢占系统级默认入口
- 把能力沉到 OS 基础设施和 agent runtime
- 建好 app 接入层和动作分发层
- 让自己的平台先成为 system-ready，再成为 ecosystem-ready

### 12.9 如果 app 的 agent 层和 OS 的 agent 层属于同一个品牌，它们应该怎么有机结合

这个问题很关键。

如果默认它们是同一个品牌，我最不建议的做法是：

- OS 里一个 agent
- app 里再做一个完全独立的 agent
- 两边各有自己的记忆、人格、任务和入口

这样用户感受到的不会是“更完整”，而是：

- 两个脑子
- 两套状态
- 两套权限
- 两套入口在打架

所以如果是同一个品牌，我认为最合理的原则应该是：


> **一个大脑，两个前台。**

也就是：

- **OS agent 层**负责系统级入口、上下文和高频轻交互
- **app agent 层**负责深 workflow、复杂编辑、历史沉淀和正式系统回写

#### 12.9.1 OS agent 层到底负责什么

OS 层不应该试图承接所有复杂工作，而应该专注于它天然有优势的部分：

**A. 入口**

- 锁屏
- 物理按钮
- 侧边键
- 语音唤起
- 相机
- 通知
- 全局分享入口

**B. 轻量上下文理解**

- 当前屏幕
- 当前 app
- 当前时间和地点
- 最近交互对象
- 基础个人上下文

**C. 快速 capture**

- 录音
- 截图
- 相机拍摄
- 短文本
- 简单问题和简单命令

**D. 快速动作**

- 记下来
- 稍后提醒
- 加一个待办
- 发起一个草稿
- 打开正确的 app 深入处理

一句话说，OS agent 更像：

> **随时在线的入口层和分发层。**

#### 12.9.2 app agent 层到底负责什么

app 层的价值不是“也能聊天”，而是把同一品牌的 agent 变成真正能交付结果的工作台。

它更适合承接：

**A. 深 workflow**

- CRM 跟进
- 工单生成
- 巡检报告
- 内容整理
- 项目任务回写

**B. 复杂编辑和确认**

- 修改结构化结果
- 补充字段
- 批量处理
- 审核和确认高风险动作

**C. 历史与记忆**

- 历史记录
- 长期记忆
- 项目上下文
- 客户上下文
- workflow 模板

**D. 系统回写**

- CRM
- 工单系统
- Notion
- Jira / Linear
- ERP / 表单 / 企业数据库

一句话说，app agent 更像：

> **同一个 agent 的深工作台和系统控制面板。**

#### 12.9.3 两层之间最重要的不是“都能做什么”，而是“如何握手”

如果是同一品牌，最核心的不是功能列表，而是 handoff 做得顺不顺。

我认为至少要有四个共享层。

#### 共享层 1：同一个身份

用户不应该感觉：

- OS 里是一个助手
- app 里是另一个助手

更好的体验应该是：

- 同一个名字
- 同一个语气
- 同一个历史
- 同一个用户偏好

#### 共享层 2：同一个任务对象

用户在 OS 层说：

- “帮我记一下刚才客户说的需求，回头整理成 follow-up”

这个任务不应该在进入 app 之后变成一件全新的事。

它应该始终是：

- 同一个 task id
- 同一个状态机
- 同一个上下文包

比如状态可以统一成：

- captured
- structured
- needs_review
- ready_to_sync
- synced
- follow_up_pending

这样 OS 和 app 就不是两套逻辑，而是在共同推进同一个对象。

#### 共享层 3：同一份记忆

如果 OS 记得：

- 你经常把客户拜访语音转成 CRM 跟进

但 app 不记得，体验就会断。

反过来也一样。

所以更合理的是：

- 共享个人偏好
- 共享 workflow 偏好
- 共享项目 / 客户 / 场景记忆

区别只在于：

- OS 读取的是轻量实时记忆
- app 能展示和编辑的是更完整的历史记忆

#### 共享层 4：同一套权限和信任模型

同品牌下最糟糕的体验之一是：

- OS 问一次权限
- app 再问一次权限
- 用户不知道到底哪个层在访问数据

更好的做法应该是：

- 用户面对的是同一个品牌权限模型
- OS 代为做轻量调用
- app 负责展示更完整的权限说明和管理

#### 12.9.4 最自然的交互路径应该是什么样

如果同品牌要把两层真的做顺，我最推荐下面这条路径：

**第一步：在 OS 层 capture**

例如：

- 长按按钮开始录音
- 拍一张现场照片
- 截一个聊天图
- 语音说一句“帮我整理成后续动作”

**第二步：OS 层做快速理解**

OS 层先做：

- 初步 ASR / OCR
- 基础上下文拼接
- 判断这是不是轻任务还是重任务

**第三步：重任务无缝进入 app**

如果需要：

- 深编辑
- 多字段确认
- 正式系统回写
- 批量处理

那就无缝打开 app，并且带着已经结构化好的草稿进入，而不是让用户重新来一遍。

**第四步：app 层完成深处理**

app 层负责：

- 补齐信息
- 选择客户 / 项目 / 门店
- 确认输出
- 正式同步系统

**第五步：OS 层接回跟踪**

一旦任务创建完成，OS 层继续承担：

- 提醒
- 通知
- 后续 capture
- 下一步 prompt

这时用户会感觉：

> **同一个 agent 在不同深度的界面里连续工作，而不是在两个产品之间来回跳。**

#### 12.9.5 最好的分工不是“OS 做浅的，app 做深的”这么简单

更准确一点，我会写成：

- **OS 层负责高频、低摩擦、情境触发**
- **app 层负责复杂、正式、可审计**

这个区分比“浅 vs 深”更稳。

因为有些动作虽然看起来不复杂，但如果它涉及：

- 正式记录
- 对外发送
- 业务系统写入
- 风险责任

那它就更应该进 app 层确认。

#### 12.9.6 一个很现实的例子：移动工作流

拿你前面关心的移动工作流举例。

假设是同品牌的 OS agent + app agent。

**场景：外勤销售拜访结束**

OS 层：

- 长按按钮录三分钟语音
- 自动转写
- 初步识别客户名、需求、异议、下一步动作

app 层：

- 打开到该客户页
- 自动填好拜访纪要草稿
- 让销售补两三个关键字段
- 一键同步 CRM，并生成 follow-up 任务

OS 层：

- 第二天提醒“你答应客户的报价还没发”

如果这条链是分裂的，体验会很差。

如果它是同一个品牌下的一套状态机，体验就会非常顺。

#### 12.9.7 一个很现实的例子：record 工作流

比如 founder / 创作者 / 投资人常见的 record workflow。

OS 层：

- 看到信息，按键录音或截图
- 直接说“这条回头整理成下一步”

app 层：

- 统一展示所有 capture
- 自动归类到项目、主题、人物
- 让用户批量决定哪些进待办、哪些进知识库、哪些发给别人

OS 层：

- 在合适时间弹出“你昨天说要处理的 3 个点”

这个场景本质上说明：

> **OS 层负责抓住瞬间，app 层负责把瞬间变成系统。**

#### 12.9.8 如果是同品牌，最容易犯的几个错误

我会特别警惕下面几个反模式。

**错误 1：两个入口争主入口**

- OS 想成为唯一入口
- app 也想成为唯一入口

最后结果是用户混乱。

**错误 2：两边各自存记忆**

- OS 记一份
- app 再记一份

结果是上下文不一致。

**错误 3：所有事都想在 OS 层完成**

这样会导致：

- UI 太挤
- 权限太重
- 复杂任务体验崩

**错误 4：所有事都强行跳 app**

这样又会失去 OS 层最宝贵的低摩擦价值。

#### 12.9.9 如果把这个问题浓缩成一句话

我会写成：

> **同品牌下，OS agent 层和 app agent 层最好的关系不是竞争，而是“同一个大脑的两种界面”：OS 负责高频捕捉、上下文和分发，app 负责深 workflow、确认和正式系统回写。**
