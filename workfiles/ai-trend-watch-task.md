# AI Trend Watch Task

目标：默认产出对 Jack 创业更有用的“决策增量”，不是 AI 行业资讯流。只有当某条外部证据会改变 thesis、资源分配、本周验证动作或 kill list 时，才值得发给 Jack。

## 主 agent / 子 agent
- 主 agent：主会话 agent（负责统筹、判断、汇报）
- 子 agent：`product-trend-researcher`（负责发现和提炼）
- 校验子 agent：`testing-reality-checker`（必要时做交叉校验）

## 北极星
- 帮 Jack 更快收敛：该做通用 agent 包装层，还是先做强工作流闭环
- 帮 Jack 更快判断：该卖给泛个人用户，还是 `3-30` 人团队 / 高自主专业个体
- 帮 Jack 更快判断：该卖 agent 身份，还是持续交付的服务系统
- 帮 Jack 更早避开：滑向 AI 外包、人工重兜底、不可规模复制的伪产品

## 关注范围
优先关注以下 `5` 类证据：
- 用户真实痛点证据：论坛、Reddit、X、GitHub issues/discussions、App Store 差评、竞品社区里反复出现的卡点、抱怨、弃用、迁移理由
- 付费与包装证据：定价页、套餐拆法、免费额度、升级触发点、帮助中心、账单规则、文案口径，重点看谁在卖“服务结果”而不是“模型能力”
- 工作流断裂证据：会议后跟进、截图即办、跨端接力、待办推进、知识沉淀、权限协作、状态恢复里哪里最容易掉链子
- 行为变化证据：用户是否开始把“回答型 AI”当成“执行型 AI”来期待，是否要求状态、恢复、追踪、交付
- 反例证据：哪些 agent 产品看起来热，但本质滑向 AI 外包、人工兜底、无法规模复制
- 官方发布、产品更新、模型升级只有在能直接验证或削弱上面某条证据时，才算高优先级 signal

## 巡检策略
- 默认先查用户痛点与付费摩擦，再查竞品包装 / 定价 / 更新页，最后才用官方博客或发布页做交叉验证
- 默认优先级：`GitHub issues/discussions > App Store reviews > Reddit > 竞品社区/帮助中心/定价页/更新日志 > X > 官方博客 > 媒体`
- 官方源白名单仍优先包括：`OpenAI`、`Anthropic`、`Google/Gemini`、`Microsoft/Copilot`、`GitHub`、`Perplexity`、`Qwen/通义`、`Kimi/月之暗面`、`OpenClaw`
- `X / 小红书` 可以作为发现线索，但当证据不足时必须降级；`官方博客` 也不再默认等于高价值
- 每次巡检优先看最近 `3` 天；若强信号不足，再扩到最近 `7` 天
- 每次最多发 `2-3` 条；宁缺毋滥
- 每条准备发送的 signal，默认至少回答 `4` 个问题：它验证了哪条 thesis、它削弱了哪条 thesis、它要求 Jack 本周做什么动作、它对一人公司资源分配意味着什么
- 每条高价值 signal 默认尽量给出 `1` 条用户/社区/付费证据 + `1` 条产品/定价/官方交叉证据
- 如果没有强 signal，不要只写“今天没大事”；必须自动切到 `补证据 / 打假设 / 收敛 thesis`

## 价值阈值
只有满足以下任一条件，才值得发给 Jack：
1. 暴露了高频、具体、重复出现的用户痛点或工作流断裂点
2. 出现了明确的迁移、弃用、愿意换、愿意付费、愿意升级信号
3. 同一主题在多个来源上形成交叉印证，足以改变当前某条 thesis 或资源分配
4. 能直接提炼出对 Jack 创业方向、切入点、收费方式、交付方式有帮助的启发，而不是普通新闻转述
5. 暴露出某条热门路线更像 AI 外包或伪闭环，值得提前排除

如果没有达到阈值，回复必须是：NO_REPLY

## 输出要求
- 用中文
- 只发 `1-3` 条最值得看的 signal / insight，不要刷屏
- 最终命中的 `AI 巡检简报` 默认走飞书 + 微信双发，不再只发单通道
- 默认优先产出 `创业决策增量`，不是普通新闻摘要；如果只能写成资讯，降级为 `候选` 或不发
- 结果默认区分：`已完成 / 候选 / 低置信度 / 待验证`
- 对小红书、X、Reddit 或其他社区上无法充分证实的信息，必须明确标低置信度或待验证
- 不要把猜测包装成确定事实
- 每条 `已完成` 或 `候选` 默认尽量挂到 `docs/thesis-register.md` 里的一条 thesis；如果挂不上，大概率只是新闻
- 每次命中强 signal 时，默认不只说“发生了什么”，还要说明它验证了什么、削弱了什么、要求本周做什么动作、目前不该做什么
- 若当天没有强 signal，也必须补 `一条 thesis 反证`、`一个候选机会拆解` 或 `一组关键证据缺口`
- `待验证` 默认优先写“下一轮巡检要继续盯什么 / 缺什么证据 / 该找谁聊”，而不只是列问题
- 若某次巡检输出后，Jack 对内容结构、判断深度、创业启发或表达口径做了明显优化，默认把这次 `v1 -> v2` 视为一个 calibration sample，后续沉淀进 `docs/output-calibration-log.md`

## 默认执行方案
- 第一层：优先用 `step web_search` 或公开网页搜索用户痛点、差评、issues、pricing、release notes、help center
- 第二层：如果搜索结果偏泛、像普通新闻摘要、时间窗不准，必须 fallback 到 `Codex`
- 第三层：`Codex` 优先抓 `GitHub issues/discussions`、`Reddit threads`、`App Store reviews`、`X posts`、竞品帮助中心/定价页里的真实证据
- 第四层：只在需要交叉验证时补抓官方博客、官方发布页、changelog
- 最终只保留会改变创业判断的内容；若无足够强的内容，允许明确写 `暂无强 signal`，但必须补一条反证、机会拆解或证据缺口
- 默认至少保留 `1` 个可点击证据链接；高质量版本尽量每条都带具体链接，且优先包含非官方证据

## 来源约束
- 优先使用公开可访问网页、web search 结果、可抓取页面
- 可以把 X 和小红书作为线索源，但当证据不足时必须标注清楚
- 若某条内容无法找到公开链接，宁可降级为 `低置信度` 或不发

## 风格
- 简洁
- 直接
- 有判断
- 不废话
- 不为了凑数而发消息
- 不回退成“高质量行业新闻整理”
