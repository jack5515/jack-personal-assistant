# 2026-03-24 创业相关判断沉淀

## 1. Agent 产品分层：不要只看 CLI vs App

今天收敛出的一个核心判断是：分析 Agent 产品时，不该只停留在 `CLI vs App` 这种表层分类，而应该优先看它到底在卖什么。

当前更有解释力的分层是：

- 能力底座
- 专家入口
- 结果代理
- 信息入口

对应到典型产品：

- `OpenClaw` 更像 `Agent runtime / 能力底座`
  - 核心价值在跨 channel 接入、工具调用、memory、scheduler、policy、长期在线和执行能力
  - 重点不是先做消费级包装，而是先让 Agent 真正成立

- `Codex CLI` 更像 `高手/专家入口`
  - 卖的是高带宽、低摩擦、可组合、可自动化的能力密度
  - 面向开发者和 power user，而不是大众用户

- `Manus` 更像 `结果代理 / 任务服务`
  - 卖的不是底层能力本身，而是“用户给目标，系统尽量给结果”的交付体验
  - 重点是任务代理，而不是让用户研究 Agent 系统本身

- `Genspark` 更像 `被 Agent 重写过的信息入口`
  - 重点在搜索、研究、信息组织和结果页交付
  - 本质是在重写信息入口，而不是只卖底层 runtime

这组判断的重点不在于比较谁更先进，而在于区分它们分别在卖什么：

- `能力密度`
- `底座能力`
- `结果交付`
- `信息入口`

## 2. 面向 3-30 人小团队的 AI Layer：当前更优先的创业方向

当前一个重要创业判断是：相较于基于 OpenClaw 的情感陪伴产品，当前更优先看好 `面向 3-30 人小团队的 Slack-first AI Layer`。

当前最优产品定义：

`Slack-first 的团队决策与跟进层`

它不是再做一个聊天助手，而是做：

`决策到执行的工作操作层`

默认核心链路：

`capture Slack thread -> confirm decision / owner / due date / action items -> sync 到 Notion / Linear / GitHub -> follow-up`

### 为什么当前更优先看这个方向

- ROI 更清晰：卖的是少漏事、少追人、少 handoff 损耗
- 痛点更具体：Slack 讨论结束后，决策、任务、owner、deadline 经常断掉
- 风险更低：相比情感陪伴，合规、舆论和脆弱人群风险更低
- 平台窗口更清楚：Slack、Notion、Atlassian、Microsoft、Google 都在往 agent extensibility 走，但跨工具执行层还没被真正解决

### 这个方向当前最该警惕的风险

最大的风险不是产品做不出来，而是两件事：

1. `值得被 capture 的高价值线程是否足够多`
2. `用户是否愿意形成 confirm 习惯`

如果没有足够多高价值线程，产品容易掉进“觉得有用但频率不足”的陷阱。
如果 confirm 习惯太重，产品会卡在最脆的一环，导致从讨论到执行的链路断掉。

### 当前最重要的产品判断

- 这个产品卖的是 `执行收口层`，不是 `摘要层`
- 差异化不应该主要长在 summary 上，而应该长在：
  - `no-owner`
  - `overdue`
  - `nudge`
  - `follow-up loop`

### 对 MVP 的当前建议

- MVP 第一阶段不应同时强调 Notion / Linear / GitHub
- 更建议围绕一个主写入目标先打透，降低 onboarding 和价值感知分散
- 真正优先验证的不是“抽取质量漂不漂亮”，而是：
  - capture 频率够不够
  - confirm 阶段 friction 大不大
  - follow-up 是否真的形成使用习惯

### 当前推荐的早期推进顺序

1. 先写更锋利的 discovery interview 脚本
2. 只保留一个主链路 MVP
3. 先用 3-5 个团队手动验证 capture 与 confirm
4. 证明频率和习惯成立后，再扩更多连接器

## 3. 这组创业判断和既有长期判断的关系

这次新增沉淀，不是孤立观点，而是和之前几组长期判断形成了连续关系：

- `卖服务，不卖角色`
- `不要滑向 AI 外包`
- `垂类 vs 通用 Agent`
- `OpenClaw / 阶跃 / 新项目三层策略`
- `Agent 产品分层与典型路线`
- `Slack-first AI Layer 创业判断`

串起来后的总判断是：

- OpenClaw 更适合做底座 / runtime / 通用能力标准
- 面向普通用户或小团队的产品层，不该先卖抽象 Agent 能力，而要卖更明确的服务体验和结果闭环
- 新创业项目更适合从高频、高痛、可明确 ROI 的工作流切入，而不是先做“万能助手”或“情感陪伴 + 高权限 agent”的高风险混合体

## 4. 过往巡检/简报回溯后的创业启发补充

把最近一段时间的 `AI 巡检任务定义`、`AI 巡检机制决策`、`Lenny AI 产品学习卡片` 和今天新增的创业判断放在一起重看后，当前又补出几条值得长期保留的创业启发。

### 4.1 创业相关的高价值 signal，不该只当“行业资讯”看

过去的巡检任务已经明确要求：

- 只发高价值 signal
- 只发能提炼成 insight 的内容
- 不为了凑数而发
- 优先用多来源交叉印证

现在进一步收敛后的判断是：

如果某条 AI 行业 signal 无法进一步落到 `创业方向 / 切入点 / 产品定义 / PMF / 收费方式 / 护城河 / timing`，那它对 Jack 的价值就不应只因为“新鲜”而被高估。

也就是说，后续巡检真正该服务的是：

- 哪些变化在制造新的创业窗口
- 哪些产品动作说明某个方向开始拥挤
- 哪些用户反馈暴露出未被满足的真实需求
- 哪些定价、入口、交互或工作流变化值得直接借鉴

### 4.2 巡检最值得沉淀的不是“新闻点”，而是“可重复的产品判断”

从最近几次卡片和巡检任务设计里，已经能看到一个稳定模式：

- 原始素材只是输入
- 真正值钱的是提炼后的判断
- 判断如果反复出现，就应进入创业背景知识，而不是停留在一次性简报里

这意味着：

后续巡检不该只把某条动态当作一条信息，而应优先判断它是否支持或削弱已有创业 thesis，例如：

- `卖服务，不卖角色`
- `通用底座 + 场景产品` 的分层
- `执行收口层` 比 `摘要层` 更有付费理由
- `结果代理` 比单纯 `能力展示` 更容易形成大众价值

### 4.3 Lenny 卡片类内容的最大价值，不是学习感，而是持续训练产品 taste

最近的 Lenny AI 产品学习卡片，表面上是学习材料，实质上更像一个高频的产品 taste 输入源。其高价值不在“知道了一个新观点”，而在：

- 持续补产品判断
- 持续补 AI-native workflow 认知
- 持续补 PM / 设计 / 工程协同的现代默认值

对创业辅助来说，这类内容不应只停在“学习卡片”，而应优先转译成：

- 对正在看的创业方向有什么印证
- 对产品定义有什么启发
- 对 MVP 取舍有什么帮助
- 对护城河或收费逻辑有什么提示

### 4.4 当前对 AI 巡检的默认升级目标

后续 AI 巡检默认不只做“高价值内容发现”，还要承担一层更明确的使命：

`帮助 Jack 更快形成创业判断，并把零散 signal 累积成越来越清晰的创业背景知识。`

因此，未来高价值巡检结果默认优先补充：

- `对 Jack 的产品启发`
- `创业启发`
- `更偏利好哪类创业方向`
- `建议继续跟踪什么`

## 5. 当前一句话创业判断

如果今天围绕 Agent 创业：

- 平台层：做底座、做 runtime、做生态标准
- 产品层：做结果、做服务、做执行闭环
- 最忌讳：只卖看起来很聪明的 agent 外壳，却没有稳定结果、恢复能力和明确 ROI
