## 记忆落点

  - 阶跃 AI = 多端 Agent 产品体系，而非单点录音工具
  - 核心竞争链路 = `context 积累 -> 记忆沉淀 -> Agent 执行 -> 市场生态 -> 商业化`

  - 普通 PM 被压缩，强 PM 被放大
  - 新标准重点不在会不会用工具，而在是否能定义价值、评测效果、设计协同、理解约束、形成闭环

- 新增功能记忆：`PRD - 阶跃AI APP 接入Openclaw`
  - 产品主线：一期先支持“绑定已有 OpenClaw”，二期主推“一键部署 OpenClaw”。
  - 入口形态：Chat 上方常驻 `Openclaw Agent`，后续扩展为 `更多Agent`。
  - 一期绑定页：给用户 `APP ID + API Key` 复制到自己的 OpenClaw，并引导切换到 `step 3.5` 与查看插件教程；绑定成功后进入 Agent 聊天页。
  - 二期一键部署：系统静默完成云端 OpenClaw 部署、默认配置 `step-flash3.5`、自动绑定实例 URL 和 API Key 到用户上下文。
  - Agent 页定位：单独聊天页，不新起会话；先用文字展示工作状态，后续再做虚拟桌面；支持状态红点、解除绑定、删除云端实例。
  - 默认推荐问题：配置企业微信机器人、配置飞书机器人、搜索 AI 热点新闻。
  - 当前更优商业方案：先用“限免 + 邀请码/白名单 + token 上限”快速上线验证，而不是先接开放平台付费或先做完整订阅。
- 新增产品论证记忆：卖的不是“智能体角色”，而是“持续服务能力”。
  - 与 `GPTs / 智能体商店` 的关键差异，不应定义为“有更多 agent/技能/人格”，而应定义为：`一个强 runtime + 面向不同用户 taste 的服务编排层`。
  - `一个牛逼的 claw 就够了` 的前提是：底层 runtime、执行链路、记忆、连接能力足够强；真正需要差异化的是服务 taste，而不是无穷无尽的人设。
  - 服务 taste 的内核包括：默认模型、默认 skill 组合、记忆策略、汇报风格、主动性、权限处理、安全策略、失败恢复、面向不同角色的默认服务预设。
  - 产品机会点不在“再做一个智能体市场”，而在于把复杂的 agent runtime 产品化，让普通用户不用理解模型/API/Skill/调试细节也能用起来。
- 新增产品红线记忆：必须明确区分“AI 服务”与“人工外包平台”。
  - 人工外包卖的是人力时间和项目交付，系统主要做撮合；Claw 型服务如果成立，卖的是 `长期在线、可持续调用、可规模复制的服务系统`。
  - 只有当服务能力能沉淀为 workflow、skill、memory、policy、service preset，并被系统重复执行时，它才是产品；否则只是披着 agent 外壳的 AI 外包。
  - 后续判断这条路线是否成立时，默认检查：是否过度依赖人工介入、是否缺少默认配置和恢复机制、是否每个用户都像在做定制项目、是否能力没有沉淀到系统里。
  - 默认产品红线：不能把核心价值建在“有人替你搞定”上，而要建在“系统默认帮你搞定，人只在边缘兜底和进化系统”上。
- 新增差异化判断：与人工外包平台的本质差异，不只是更快，而是 `把好的交付抽象成可配置、可复制、可售卖的 taste 参数`。
  - 人工外包卖的是他人的时间；这里如果成立，卖的是 `可复制的品味`。
  - 关键差异有三层：即时交付而不是排期等待、边际成本接近软件而不是线性消耗人力、系统会累积并复用 taste 而不是每次重新对齐需求。
  - 一个好比喻是：好厨师一次只能服务一桌，但沉淀出的菜谱可以开连锁店。对应到产品上，核心问题不是能不能交付，而是能不能把“好的交付”沉淀成可参数化、可模板化、可规模售卖的服务配方。
  - 后续判断这条路线的护城河时，默认检查：交付质量是否仍然强依赖人工判断；如果是，那它和外包平台的区别就只剩速度，护城河不成立。
